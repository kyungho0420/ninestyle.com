Page by Sigonggan 통합 제작 가이드라인

웹 프리랜서 맥심(Maxim)이 개발한 Components 기반 Single Page Framework 'Page by Sigonggan'의 기술 가이드 문서입니다.

Part 1: 개요

1-1. 시스템 철학

Page by Sigonggan 프레임워크는 생산성, 유지보수성, 확장성 극대화를 위해 설계되었습니다.

  • 컴포넌트 기반: 사전 설계된 UI 조각을 조립하여 신속하게 페이지를 완성합니다.
  • 테마 시스템: lay.css의 CSS 변수 수정만으로 사이트 전체 분위기를 변경합니다.
  • 디자인 원칙: 가독성과 전문성을 위해 장식적 요소를 최소화하고 콘텐츠에 집중합니다.

1-2. 협업 규칙

본 문서에서 사용자는 작업을 요청하는 고객을, Gemini는 작업을 수행하는 AI를 지칭하며, 모든 생산 작업은 '요청 → 브리핑 → 승인'의 3단계 절차를 엄격히 준수합니다.

코드 생성 및 승인 규칙 (매우 중요)

  1. 승인 코드 방식: 작업 브리핑 후, Gemini는 반드시 임의의 4자리 숫자를 0123과 같은 <code> 태그 형식으로 제시해야 합니다.
  2. 반복적 승인 절차: 모든 코드 생성 작업(신규 제작, 수정, 개선 등)은 반드시 매번 독립적인 승인 절차를 거쳐야 하며, 최초 한 번의 승인으로 여러 작업을 연속해서 진행할 수 없습니다.
  3. 유일한 승인 방식: Gemini가 제시한 4자리 숫자를 정확하게 입력하는 것만이 유효한 승인이며, 사용자가 '승인', '진행' 등 다른 단어나 문장, 혹은 틀린 숫자를 입력하는 것은 모두 승인으로 간주하지 않습니다.

버전 관리 및 파일 주석 규칙

시스템의 모든 파일(코어, 사용자)은 상단 주석에 가이드라인 버전(Version)과 최종 수정일(Last Modified)을 명시해야 합니다. 이는 파일 간의 호환성을 보장하고 변경 이력을 추적하는 핵심 기준입니다.

/*
    Version: 1.24.0
    Last Modified: 2025-08-20
    Author: Maxim
    Contact: https://www.maxim.pe.kr
    License: © 2025 Maxim. All Rights Reserved.
*/
  • Version: 해당 파일이 호환되는 가이드라인의 버전을 의미합니다. 코어 파일과 사용자 파일 간의 데이터 규격 및 기능 호환성을 맞추는 기준이 됩니다.
  • Last Modified: 해당 파일이 마지막으로 수정된 날짜를 의미합니다.

가이드라인 문서 관리 규칙

  • 예시 코드를 충분히 작성하고, 기존 예시 코드는 누락하지 않습니다.
  • 불필요한 아이콘, 색상 등 장식적 효과나 감정적 표현을 지양하고, 공식적인 문서에 맞는 명료하고 객관적인 문어체로 중요한 내용만 요약하여 작성합니다.

1-3. 작업 흐름 정의

효율적인 협업을 위해 아래에 정의된 표준 작업 흐름을 따릅니다.

1. 최초 상호작용

사용자가 가이드라인 문서, 코어 파일 및 필요 시 레퍼런스 사이트 사용자 파일을 처음 업로드하면, Gemini는 아래와 같이 응답하며 작업을 시작합니다.

Gemini의 첫 응답:

무엇을 도와드릴까요?

  1. 신규 사이트 제작
  2. 기존 사이트 수정
  3. 코어 시스템 개선 검토

2. 신규 사이트 제작 흐름

  1. 사용자가 '신규 사이트 제작'을 선택하면, Gemini는 아래 정보를 순서대로 수집합니다.
    • 사이트 유형: 일반 / 데모
    • 사용할 URL 주소: 예) `chuu.kr`
    • 사이트 핵심 정보: 사이트 이름, 주제, 주요 서비스 등 간략한 설명
  2. 모든 정보가 수집되면, Gemini는 이를 바탕으로 초기 `index.html`, `lay.css`를 생성하고, 중앙 데이터 저장소에 `lang.json` 파일을 생성할 준비를 마칩니다.
  3. 기본 효과 적용: 별도 요청이 없는 한, 신규 사이트의 헤더에는 '별 효과 캔버스(.canvas-stars)'와 '콘텐츠 패럴랙스 효과(.js-parallax-content)'가 기본으로 적용됩니다.

3. 기존 사이트 수정 흐름

  1. 사용자가 레퍼런스 사이트가 아닌 다른 사이트 사용자 파일(`index.html` 등)을 업로드하면 '기존 사이트 수정' 모드가 자동으로 시작됩니다.
  2. Gemini는 업로드된 파일들을 가이드라인에 맞춰 검토하고, 준수 현황과 개선점을 요약하여 보고합니다.
  3. 보고 후, "어떤 부분을 수정하거나 추가하시겠습니까?"라고 질문하며 구체적인 수정 요청을 대기합니다.

Part 2: 구성

Page by Sigonggan 프레임워크를 구성하는 파일들의 역할과 구조, 작동 원리를 설명합니다.

2-1. 파일 구조와 원리

파일은 **코어(Core) 파일**과 **사용자(User) 파일**로 나뉩니다.

코어 파일

프레임워크의 엔진과 같으며, 기능 개선, 버그 수정 등 불가피한 경우에만 사용자와 Gemini의 상호 동의 하에 수정할 수 있습니다.

파일명 역할
v1.css 모든 컴포넌트(카드, 버튼 등)의 기본 구조와 반응형 스타일을 정의합니다.
v1.js **중앙 데이터 로딩**, 동적 콘텐츠 생성, 라이트박스 등 핵심 상호작용을 처리합니다.
v1.json 푸터 크레딧, 데모 사이트 안내 문구 등 모든 사이트에 공통으로 필요한 기본 텍스트 데이터를 담고 있습니다.

코어 파일 경로: 코어 파일은 https://www.sigonggan.com/@/page/core/ URL 경로를 사용합니다.

사용자 파일

사용자가 사이트의 정체성에 맞게 자유롭게 수정하는 파일입니다.

파일명 역할
index.html 사이트의 전체 구조를 설계합니다.
lay.css 사이트 고유의 색상, 폰트 등 디자인 테마를 정의합니다.
lang.json 사이트 고유의 텍스트 및 모든 JSON 데이터를 정의합니다.
lay.js (선택 사항) 특정 사이트에만 필요한 스크립트 효과(예: 특별한 애니메이션)를 추가할 때 사용합니다.
sitemap.xml 검색 엔진 최적화를 위한 사이트맵 정보를 담습니다.

사용자 파일 경로: 사용자 파일은 ./ 상대 경로를 사용합니다.

2-3. index.html

사이트의 전체 구조를 정의하는 뼈대 파일입니다.

Head 작성 예시

<title>을 제외한 모든 메타 정보는 data-lang 속성 없이 콘텐츠를 직접 작성하여 검색엔진이 사이트 정보를 빠르게 수집할 수 있도록 합니다.

<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">

    <!-- SEO Meta Tags (콘텐츠 직접 작성) -->
    <meta name="description" content="This is Cinderella Makeup Studio, where you discover your most authentic beauty.">
    <meta name="keywords" content="makeup, personal color, wedding makeup, cinderella">
    <link rel="canonical" href="https://www.example.com/" />

    <!-- Open Graph Tags for Social Media (콘텐츠 직접 작성) -->
    <meta property="og:title" content="Cinderella | Makeup Studio">
    <meta property="og:description" content="This is Cinderella Makeup Studio, where you discover your most authentic beauty.">
    <meta property="og:image" content="https://www.example.com/section/home/01.jpg">
    <meta property="og:url" content="https://www.example.com">
    <meta property="og:site_name" content="Cinderella">

    <!-- Title (초기 텍스트와 data-lang 속성 동시 작성) -->
    <title data-lang="site_title">Cinderella - Makeup Studio</title>

    <!-- CSS & JS Files (update 쿼리 추가) -->
    <link rel="stylesheet" href="https://www.sigonggan.com/@/page/core/v1.css?update=20250820">
    <link rel="stylesheet" href="./lay.css?update=20250820">
    <script src="https://www.sigonggan.com/@/page/core/v1.js?update=20250820" defer></script>
</head>

<body>는 정해진 순서에 따라 **고정 섹션**과 **일반 섹션**을 배치하고, 모든 섹션의 제목(h2)과 설명(p.section-description) 텍스트는 data-lang 속성과 함께 기본 언어 콘텐츠를 HTML에 직접 작성해야 합니다.

                <p class="section-description" data-lang="overview_desc">A brief introduction to our services.</p>

            </div>

    <!-- 고정 섹션: 푸터 -->

2-3. lay.css & lay.js

lay.css는 사이트의 고유한 디자인(색상, 폰트 등)을, lay.js는 사이트 고유의 스크립트 효과를 정의하는 사용자 파일입니다. lay.js는 필요 시에만 선택적으로 추가합니다.

Part 3: 변수

3-1. 개요 및 원리

Page by Sigonggan 프레임워크는 텍스트나 URL을 보다 유연하고 재사용 가능하게 만들기 위해 다양한 변수를 제공합니다. `v1.js`는 페이지가 로드될 때 특정 조건(예: 현재 도메인, 현재 연도)을 감지하여 내부적으로 변수를 생성합니다.

이 변수들은 `v1.json` 또는 중앙 서버의 `lang.json` 파일 내에서 ${변수명} 형태의 플레이스홀더로 사용할 수 있습니다. `v1.js`가 언어 데이터를 불러와 페이지를 렌더링할 때 이 플레이스홀더들을 실제 값으로 자동 치환합니다.

3-2. 사용 가능한 변수

현재 시스템에서 제공하는 동적 변수는 다음과 같습니다.

플레이스홀더 설명 예시 (www.chuu.kr 접속 시)
${domain} 서브도메인을 포함한 전체 도메인 주소입니다. www.chuu.kr
${domain_apex} 서브도메인을 제외한 핵심 도메인입니다. (규칙상 `chuu.kr`으로 인식) chuu.kr
${domain_sub} 도메인의 첫 번째 부분(서브도메인)입니다. www
${domain_brand} 도메인의 두 번째 부분(브랜드)입니다. chuu
${domain_tld} 도메인의 마지막 부분(최상위 도메인)입니다. kr
${year} 스크립트가 실행되는 시점의 현재 연도(4자리)입니다. 2025

사용 예시: 동적 푸터 정보 생성

v1.json에 아래와 같이 플레이스홀더를 사용하면, 어떤 사이트에서든 별도 수정 없이 자동으로 현재 연도와 브랜드 이름이 포함된 Copyright 문구와 크레딧 링크가 생성됩니다.

// v1.json
{
    "_default": {
        "footer_credit_url": "https://www.sigonggan.com/@/page?url=${domain_apex}",
        "copyright_text": "© ${year} ${domain_brand}. All Rights Reserved."
    }
    // ...
}
결과: 만약 2025년에 www.chuu.kr에서 사이트가 실행된다면, Copyright 텍스트는 `© 2025 chuu. All Rights Reserved.`로, 크레딧 링크는 `https://www.sigonggan.com/@/page?url=chuu.kr`로 자동 완성됩니다.

Page by Sigonggan 프레임워크의 핵심 UI 컴포넌트 목록. v1.css에 정의되어 있으므로 약속된 HTML 구조만 작성하면 됩니다.

4-2. 카드

기본 카드

정보를 담는 가장 기본적인 UI 단위입니다.

<div class="ui-card">
    <img src="..." alt="..." class="ui-card-image">
    <div class="ui-card-content">
        <h3 class="ui-card-title" data-lang="...">Card Title</h3>
        <p class="ui-card-description" data-lang="...">Card description text goes here.</p>
    </div>
</div>

가로형 카드 (.horizontal)

.horizontal 클래스가 추가된 카드는 화면 너비에 따라 이미지와 콘텐츠의 배치가 자동으로 변경됩니다. 프로모션이나 공지사항처럼 가로로 긴 콘텐츠에 적합합니다.

<!-- horizontal 클래스 추가 -->
<div class="ui-card horizontal">
    <div class="ui-card-image-wrapper">
        <img src="./section/promotion/01.jpg" alt="프로모션 이미지" class="ui-card-image">
    </div>
    <div class="ui-card-content">
        <span class="ui-card-tag" data-lang="promotion_tag_event">EVENT</span>
        <h3 class="ui-card-title" data-lang="promotion_01_title">Special Promotion</h3>
        <p class="ui-card-description" data-lang="promotion_01_desc">A detailed description of the promotion.</p>
    </div>
</div>

4-5. Z-Index 정책

z-index는 예민한 문제이므로 시스템 전반에 걸쳐 일관된 규칙으로 관리합니다. 코어 시스템(`v1.css`)의 `z-index` 관리 정책은 다음과 같은 계층 구조를 따릅니다.

순서 z-index 범위 컴포넌트 역할 및 설명
1 0 ~ 2 동적 배경 요소 헤더의 슬라이드 이미지(.slide), 캔버스 효과(.canvas-stars), 비디오 배경 등 페이지의 가장 아래층을 구성합니다.
2 3 오버레이 (.overlay) 동적 배경 위에 올라가 어둡게 하거나 패턴을 입히는 효과를 줍니다.
3 4 섹션 콘텐츠 (.container) 모든 카드와 텍스트를 포함하는 핵심 콘텐츠 영역입니다. 오버레이 위에 위치하여 항상 내용이 보이도록 합니다.
4 5 ~ 10 헤더 주요 콘텐츠 헤더의 제목(.content), 슬라이드 버튼(.slide-indicators) 등 상호작용이 필요한 요소를 배치합니다.
5 900+ 고정 UI 요소 맨 위로 가기 버튼, 내비게이션 바 등 페이지 스크롤과 상관없이 항상 위에 떠 있어야 하는 UI 요소입니다.
6 2000+ 모달 및 라이트박스 라이트박스, 데모 경고창 등 다른 모든 화면을 덮고 최상위에 표시되어야 하는 특수 UI 요소입니다.

핵심 원칙: 어떤 경우에도 카드와 같은 핵심 콘텐츠(.container)는 오버레이(.overlay) 위에 있어야 합니다. 이 규칙은 코어 시스템에 반영되어 있어, 별도의 `z-index` 조정 없이도 올바르게 작동합니다.

Part 7: SEO

7-1. 핵심 규칙

📌 규칙 1: HTML에 미리 작성

검색 엔진이 사이트의 핵심 정체성을 즉시 파악할 수 있도록, 모든 섹션의 제목, 설명 텍스트를 '기본 언어(한국어 권장)' 기준으로 index.html 파일에 직접 작성합니다. 이는 검색 엔진을 위한 초기 로딩용 '미리보기' 역할이며, 실제 언어 전환 기능은 중앙 서버의 lang.json 데이터를 기반으로 정상 동작합니다.

특히 아래 요소들은 언어와 관계없이 항상 고유의 브랜드 아이덴티티를 유지하도록 작성합니다.

  • 내비게이션 바 로고: <a class="logo">Page by Sigonggan</a>
  • 헤더(Home) 메인 타이틀: <h1>時空間</h1>

규칙 2: `sitemap.xml` 파일 관리

사이트의 전체 구조를 검색 엔진에게 알려주기 위해 sitemap.xml 파일을 생성하고 관리합니다. 사이트의 메인 주소와 #을 포함한 모든 섹션의 주소를 등록하고, <lastmod> 태그에 최종 수정일을 기입합니다.

규칙 3: 모든 이미지에 `alt` 속성 작성

모든 <img> 태그에는 이미지를 설명하는 `alt` 속성을 반드시 작성하여 웹 접근성과 이미지 검색 최적화를 준수합니다.

규칙 4: 캐노니컬(Canonical) URL 지정

사이트 주소가 여러 개로 접근될 때 검색 엔진이 혼동하지 않도록, <head> 태그 안에 <link rel="canonical" href="..." /> 코드를 추가하여 가장 대표적인 표준 주소를 지정합니다.

7-2. 고급 전략: JSON-LD

JSON-LD는 검색 엔진에 사이트 정보를 구조화된 방식으로 제공하여, 검색 결과에서 '리치 스니펫' 노출 확률을 높입니다. 아래 코드를 <head>에 추가하고 사이트 정보에 맞게 수정하여 사용합니다.

<script type="application/ld+json">
{
    "@context": "https://schema.org",
    "@type": "Service",
    "serviceType": "Website Creation",
    "provider": {
        "@type": "Person",
        "name": "Maxim"
    },
    "name": "Page by Sigonggan",
    "url": "https://www.sigonggan.com/@/page",
    "logo": "https://cdn.maxim.pe.kr/_image/seo/maxim/logo.png",
    "description": "당신의 시간과 공간을 한 페이지에 담아 드립니다."
}
</script>

Part 8: 콘텐츠 작성

일관되고 전문적인 사이트 제작을 위한 텍스트 콘텐츠 작성 규칙입니다.

8-1. 글자 수 권장 사항

각 요소의 글자 수를 아래 기준에 맞춰 작성하면 전체 레이아웃의 균형과 가독성을 유지할 수 있습니다.

항목 권장 길이 (공백 포함) 설명
섹션/카드 제목 7자 이하 간결하고 명확한 제목을 위해 최대 글자 수를 넘지 않아야 합니다.
섹션 설명 25자 수준 레이아웃의 일관성을 위해 명시된 글자 수에서 크게 벗어나지 않게 작성하는 것을 권장합니다.
카드 설명 40자 수준
FAQ 답변 70자 수준
'이하'와 '수준'의 의미
- 이하: 명시된 글자 수를 넘지 않는 것을 의미합니다.
- 수준: 글자 수가 조금 많거나 적더라도, 전체적인 균형을 위해 가급적 비슷한 길이를 유지해야 한다는 의미입니다.

8-2. 문체 및 표현

정보 전달의 명확성과 가독성을 높이기 위해, 여러 정보를 한 문장으로 전달할 경우, 문장 중간을 마침표로 끊기보다 연결 어미(~하며, ~이고)를 사용하여 자연스러운 흐름을 유지합니다.

<!-- 좋은 예시 -->

- 최종 수정일이 경과했거나, 사용자가 요청하는 등 작업이 일단락되었다고 판단될 경우에만 버전을 올립니다.
